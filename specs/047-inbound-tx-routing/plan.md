# Implementation Plan: Inbound Transaction Routing & User Notification

**Branch**: `047-inbound-tx-routing` | **Date**: 2026-03-02 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/047-inbound-tx-routing/spec.md`

## Summary

Route incoming peer transactions to local users and notify them of pending actions. Register Service maintains a Redis-backed bloom filter of local wallet addresses. When transactions are stored, recipient addresses are checked against the bloom filter. Matches trigger gRPC calls to Wallet Service, which resolves address → wallet → user, checks notification preferences, and delivers real-time notifications via SignalR or queues for digest batching. Recovery mode detects docket gaps on startup and catches up via streaming gRPC from Peer Service, delivering missed notifications. New gRPC contracts define 8 RPCs spanning Register, Wallet, and Peer services. Notification preferences (method + frequency) extend the Tenant Service user profile.

## Technical Context

**Language/Version**: C# 13 / .NET 10.0
**Primary Dependencies**: StackExchange.Redis (bloom filter + digest queue), Grpc.Net 2.71.0 (6 new RPCs), SignalR (real-time push via EventsHub), .NET Aspire 13.0.0 (orchestration), FluentValidation 11.10.0
**Storage**: Redis (bloom filter bit-array, digest queue sorted sets, recovery state hashes), PostgreSQL (notification preferences via Tenant EF Core), MongoDB (transactions/dockets — existing, read-only for this feature)
**Testing**: xUnit + FluentAssertions + Moq (target >85% coverage for new code)
**Target Platform**: Linux containers (Docker), orchestrated via .NET Aspire
**Project Type**: Microservices (existing — changes span 5 services)
**Performance Goals**: <5s real-time notification delivery (SC-001), 100 dockets/sec recovery throughput (SC-004), <30s bloom filter rebuild for 100K addresses (SC-002), <0.1% false positive rate (SC-003)
**Constraints**: Privacy-preserving (bloom filter never exposed to peers), full replicas only (all transactions stored), rate limit 10 notifications/min/user during bursts (SC-010)
**Scale/Scope**: Up to 100,000 local addresses, burst processing during recovery (thousands of transactions)

## Complexity Tracking

> No constitution violations requiring justification. All changes follow existing patterns.

| Aspect | Approach | Rationale |
|--------|----------|-----------|
| Bloom filter in Redis | Raw SETBIT/GETBIT (not Redis Stack) | Avoids Redis Stack infrastructure dependency; standard Redis already deployed |
| Recovery as IHostedService | RegisterRecoveryService runs on startup | Follows existing AdvertisementResyncService pattern in Register Service |
| Digest worker in Wallet Service | NotificationDigestWorker (BackgroundService) | First hosted service in Wallet Service; closest to user/notification data |
| EventsHub wiring | Bridge in Blueprint Service | EventsHub lives in Blueprint Service; minimal coupling via IHubContext |
| Preference storage | Two new columns on existing UserPreferences | Avoids new entity/table overhead; PATCH endpoint already supports partial updates |
